

# Squix — OS для AI-агентов

## Общее описание

Squix — это система управления и оркестрации AI-моделей и агентов, которая позволяет пользователю создавать собственные “AI-команды” (или даже “AI-компании”) из разных моделей (локальных и облачных), распределять между ними задачи, управлять лимитами, стоимостью и контролировать весь процесс выполнения.

Основная идея — не использовать одну нейросеть для всего, а объединить множество ИИ в единую управляемую систему, где каждый агент выполняет свою роль.

---

## Основная концепция

Squix работает как “мозг с щупальцами”:

- центр (core) — принимает задачи и управляет системой
- щупальца — это агенты (разные роли)
- каждая “рука” выполняет свою функцию
- система следит за ресурсами, лимитами и качеством

---

## Ключевые возможности

### 1. Подключение моделей

Поддержка разных источников:

- OpenRouter (доступ к множеству моделей)
- прямые API (OpenAI, Anthropic и др.)
- локальные модели (через Ollama или аналогичные)
- кастомные endpoints

Для каждой модели задаются:

- стоимость (за токены)
- лимиты (дневные / месячные)
- скорость
- контекст
- специализация (код, текст, веб, анализ и т.д.)
- приоритет использования

---

### 2. Система агентов (ролей)

Пользователь может создавать агентов с разными ролями:

- Orchestrator (главный управляющий)
- Planner (разбивает задачи на этапы)
- Coder (пишет код)
- Debugger (ищет и исправляет ошибки)
- Reviewer (проверяет качество)
- Researcher (ищет информацию)
- Writer (пишет документацию)
- Memory Manager (работает с памятью)

Каждый агент:

- использует одну или несколько моделей
- имеет инструкции (system prompt)
- может вызывать другие агенты
- работает в рамках своей зоны ответственности

---

### 3. Оркестрация и правила (Policy Engine)

Система правил определяет:

- какую модель использовать для какой задачи
- когда использовать дешевые модели
- когда эскалировать на мощные модели
- fallback при ошибках или лимитах
- маршрутизацию между агентами

Примеры:

- “простые задачи → дешевые модели”
- “если не уверен → отправить на более мощную модель”
- “код → сначала coder, потом debugger, потом reviewer”

---

### 4. Управление лимитами и стоимостью

- учет использования токенов
- стоимость по каждой модели
- лимиты (дневные / месячные)
- предупреждения при приближении к лимиту
- автоматическое переключение на другие модели

---

### 5. Наблюдаемость (Observability)

Полная прозрачность работы системы:

- лог всех действий
- кто какой агент выполнял задачу
- какие модели использовались
- сколько это стоило
- какие были промежуточные результаты

Интерфейс должен показывать:

- цепочку действий (flow)
- взаимодействие агентов
- статус задач

---

### 6. Память и состояние

Squix должен сохранять:

- состояние проекта
- историю задач
- решения
- артефакты (код, файлы, результаты)

Типы памяти:

- краткосрочная (текущая задача)
- долгосрочная (проект)
- knowledge base

Возможность:

- продолжить работу после перезапуска
- восстановить предыдущие шаги

---

### 7. Рабочие пространства (Workspaces)

Для задач:

- изолированная работа с файлами
- доступ к проектной папке
- выполнение кода
- хранение результатов

Особенно важно для:

- разработки (код)
- работы с документами
- генерации файлов

---

### 8. UI / Панель управления

Основной интерфейс должен включать:

- список проектов
- список агентов
- список моделей
- текущие задачи
- логи и история
- стоимость и лимиты

Визуализация:

- граф задач (как flow)
- взаимодействие агентов
- статус выполнения

## Архитектура (высокоуровнево)

### 1. Core / Company Engine

- принимает задачи от пользователя
- управляет всей системой агентов
- отслеживает состояние проекта
- принимает решения на уровне всей “AI-компании”

---

### 2. Built-in Agent Team

- набор заранее подготовленных агентов
- каждый агент имеет свою роль и инструкции
- агенты уже умеют взаимодействовать между собой
- могут передавать задачи друг другу

---

### 3. Custom Agent Generator

- создает новых агентов по запросу пользователя
- генерирует для них:
  - роль
  - инструкции
  - правила взаимодействия
- подключает их к существующей системе

---

### 4. Model Registry

- список всех доступных моделей
- хранит информацию:
  - цена
  - лимиты
  - скорость
  - возможности
- используется для выбора модели под задачу

---

### 5. Resource / Limit Manager

- следит за использованием токенов
- управляет лимитами
- контролирует стоимость
- балансирует нагрузку между моделями
- решает, когда использовать дорогие модели

---

### 6. Policy Engine

- определяет, какой агент должен выполнять задачу
- выбирает модель для агента
- управляет fallback
- решает, когда эскалировать задачу
- может инициировать создание нового агента

---

### 7. Memory Layer

- хранит состояние системы
- хранит историю задач и решений
- хранит агентов и их инструкции
- позволяет восстанавливать работу

---

### 8. Observability Layer

- логирует все действия системы
- показывает взаимодействие агентов
- отображает стоимость и использование
- помогает дебажить систему

---

### 9. Workspace Layer

- работа с файлами и проектами
- выполнение кода
- хранение артефактов
- изоляция задач

---

### 10. UI Layer

- отображение всей системы
- управление пользователем
- визуализация агентов и задач
- контроль лимитов и ресурсов

# Squix

## Что это такое

Squix — это система оркестрации AI-моделей и готовых AI-агентов, которая работает как единая AI-компания внутри проекта. Она должна запускаться примерно так же, как Claude Code: пользователь открывает терминал в нужной директории проекта, запускает Squix, и дальше система начинает работать с этим проектом как с рабочей средой.

Squix должен уметь:

- подключать разные ИИ
- распределять их по ролям
- понимать, какие модели сильнее, какие дешевле, какие лучше подходят под разные задачи
- учитывать лимиты, особенно у платных моделей
- сохранять прогресс работы
- восстанавливаться после сбоев
- позволять менять конфигурацию системы под конкретный проект
- позволять создавать дополнительных агентов под задачу
- позволять отключать ненужных агентов

Главная идея проекта: пользователь не должен вручную собирать всю агентную систему с нуля. В Squix уже есть готовая структура агентов, их роли, их связи и логика взаимодействия. Пользователь запускает систему, а дальше она работает как цельная компания. При этом пользователь может менять конфигурацию, добавлять новых агентов или создавать отдельную конфигурацию только под конкретный проект.

---

## Общая идея работы

Внутри Squix уже есть набор встроенных агентов. Каждый агент знает свою роль, понимает роли других агентов и умеет передавать задачи дальше по нужной цепочке. Но в целом всей работой по задачам должен управлять orch — оркестратор. При этом orch не должен сам придумывать всю структуру работы сверху вниз: им управляет planner, который разбивает большую задачу на шаги и передает orchestration по пунктам.

То есть логика такая:

- пользователь ставит общую задачу
- planner разбивает ее на пункты
- orch получает эти пункты и распределяет между нужными агентами
- агенты общаются между собой по заранее заданной схеме связей
- каждый агент ведет свое состояние и прогресс
- вся система может быть восстановлена, если что-то сломалось или закончились лимиты

## Подробное описание структуры

### `squix/core/`

Это главное ядро работы Squix.

Оно должно:

- запускать всю систему
- загружать конфигурацию
- поднимать агентов
- связывать агентов, модели, память и policy
- следить за состоянием всей системы
- управлять жизненным циклом проекта
- обеспечивать запуск Squix из терминала в нужной директории

Именно core должен собирать всю систему в одно целое.

---

### `squix/agents/`

Здесь находятся все агенты и логика распределения задач между ними.

Эта папка отвечает за:

- встроенных агентов
- связи между агентами
- запуск агентов
- передачу задач между агентами
- расширение системы новыми агентами

#### `squix/agents/built_in/`

Здесь находятся агенты, которые изначально есть в системе.

Пока нужны такие агенты:

- `orch` — оркестратор, главный управляющий по задачам
- `plan` — planner, управляет orch и дает ему задачи по пунктам
- `build` — один или несколько builder-агентов для работы над реализацией
- `debug` — один или несколько debugger-агентов для поиска и исправления проблем
- `web` — агент для поиска информации в интернете
- `DB` — агент для работы с базой данных или знаниями проекта
- `AI` — агент для задач, связанных с ИИ, обучением, ML, моделями и всем похожим
- `idea` — агент для обсуждения идеи проекта, доработки идеи, предложения новых вариантов
- `product` — агент, который помогает превращать идею в продукт, улучшать ее, дорабатывать логику, думать о применении и о том, как это должно выглядеть как продукт
- `README` — агент, который пишет README, инструкции, описание использования и прочую документацию

Пока этого набора достаточно, потом можно будет добавить еще агентов.

У встроенных агентов уже должна быть заранее задана структура связи между собой.

### Схема связей встроенных агентов

- `idea` → `product`, `web`
- `product` → `web`, `DB`, `plan`, `orch`
- `web` → `product`, `DB`, `orch`, `idea`
- `plan` → `product`, `orch`
- `DB` → `web`, `product`, `orch`
- `orch` → `AI`, `README`, `debug`, `build`, `plan`, `product`, `web`, `DB`
- `build` → `debug`, `AI`, `orch`
- `debug` → `orch`, `build`, `AI`
- `AI` → `orch`, `build`, `debug`
- `README` → `orch`

Все они должны понимать роль друг друга, чтобы правильно распределять задачу.

Но главная логика управления должна быть такая:

- planner управляет оркестратором
- оркестратор управляет всеми остальными агентами по задачам
- остальные агенты могут общаться между собой только в рамках своей схемы связей и общей логики системы

#### `squix/agents/generator/`

Этот модуль нужен для изменения состава агентов.

Пользователь должен иметь возможность:

- увеличить количество builder-агентов
- увеличить количество debugger-агентов
- убрать ненужных агентов в конкретном проекте
- добавить своего кастомного агента в связку
- создать отдельный набор агентов только под один проект

Generator должен не просто создавать нового агента, но и:

- задать ему роль
- встроить его в общую систему
- определить, с кем он может общаться
- обновить конфигурацию связей
- обновить policy
- обновить память и наблюдаемость

---

### `squix/models/`

Здесь должна быть логика моделей и распределения моделей по агентам.

Этот модуль должен отвечать за:

- какую модель использует каждый агент
- какая модель сильнее
- какая модель дешевле
- какая модель лучше подходит под конкретные задачи
- как распределять нагрузку между моделями
- как учитывать лимиты
- как подключать платные ИИ
- как подключать бесплатные ИИ
- как подключать локальные модели

Особенно важно:

- учитывать лимиты платных ИИ
- использовать сильные модели только там, где это реально нужно
- держать дешевые или бесплатные модели для массовых и рутинных задач
- уметь переключаться, если лимит закончился или модель недоступна

---

### `squix/policy/`

Это политика общения между агентами и правила поведения системы.

Здесь должно быть описано:

- как кто с кем говорит
- какие задачи есть у каждого агента
- кому какие задачи можно передавать
- как orch распределяет задачи
- как plan управляет orch
- как обрабатывать ошибки
- как делать fallback
- как подключать новых агентов
- как менять поведение системы под конкретный проект

То есть policy — это не только правила общения, но и общая логика маршрутизации работы.

---

### `squix/memory/`

Это память и состояние системы.

Здесь должно храниться:

- состояние каждого агента
- общий прогресс работы
- текущее состояние задач
- промежуточные результаты
- история решений
- runtime-прогресс каждого агента

Это обязательно нужно для восстановления.

Каждый агент должен вести свой прогресс во время выполнения, чтобы систему можно было восстановить, если:

- закончился лимит
- какая-то модель перестала отвечать
- произошел сбой
- система перезапустилась
- пользователь прервал выполнение и хочет продолжить позже

Memory должна хранить не только итог, но и состояние процесса.

---

### `squix/observability/`

Этот модуль должен:

- логировать все действия системы
- показывать взаимодействие агентов
- отображать стоимость использования
- отображать использование лимитов
- помогать дебажить систему
- показывать, где произошла ошибка
- показывать, кто сейчас чем занимается

Это очень важный модуль, потому что без него система будет как черный ящик.

---

### `squix/workspace/`

Этот модуль отвечает за рабочую среду проекта.

Он должен включать:

- работу с файлами и проектами
- выполнение кода
- хранение артефактов
- изоляцию задач

Squix должен работать в директории проекта, как инструмент для разработки и управления задачами внутри проекта. Поэтому workspace — это место, где система реально видит файлы проекта, меняет их, создает результаты и хранит артефакты работы.

---

### `squix/api/`

Этот модуль нужен для подключения к ИИ и их API.

Здесь должно быть:

- как обращаться к каждой ИИ
- какие API доступны
- как подключать OpenRouter
- как подключать локальные модели
- как подключать прямые API разных провайдеров
- как работать с ключами, лимитами и настройками

Этот слой должен дать единый интерфейс для всех моделей, даже если у них разные способы подключения.

---

### `squix/ui/`

Это интерфейс работы системы.

Здесь должно быть:

- отображение того, как агенты общаются между собой
- отображение того, какая сейчас у каждого стадия работы
- отображение проблем
- отображение того, кто сейчас решает проблему
- отображение лимитов и стоимости
- отображение прогресса проекта

### `squix/main.py`

Это основной код запуска.

Он должен:

- запускать Squix из терминала
- определять текущую директорию проекта
- поднимать нужную конфигурацию
- запускать core
- запускать агентов
- подключать модели
- восстанавливать работу при необходимости

## Как должен работать Squix

Squix должен быть инструментом, который вызывается в терминале в нужной директории проекта.

Примерно как Claude Code, только со своей системой агентов.

Он должен уметь:

- работать в текущем проекте
- создавать агентов только для этого проекта
- менять полную конфигурацию
- отключать и включать агентов
- управлять модельным составом
- менять правила работы
- сохранять и загружать состояние

То есть это не просто библиотека, а полноценная система управления проектной AI-командой.

- кто кем управляет
